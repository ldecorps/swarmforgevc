# docs-tree.json schema

BL-117: `docs-tree.json` is a stable, versioned data contract published
alongside `backlog.json` (see `docs/backlog-dashboard-schema.md`) by
`.github/workflows/backlog-dashboard.yml`. It is generated by
`node out/tools/generate-docs-tree.js` (source:
`extension/src/docs/docsTree.ts`) and powers the PWA's documentation
explorer (vision → milestone → ticket → Gherkin).

Additive evolution only within `schemaVersion` 1, same rule as
`backlog.json`.

## Top level

| Field | Type | Notes |
|---|---|---|
| `schemaVersion` | number | Currently `1`. |
| `generatedAtIso` | string (ISO 8601) | When this file was generated. |
| `sourceSha` | string \| null | The commit this projection was generated from. |
| `vision` | `VisionDoc[]` | The product-level documents and diagrams. |
| `milestones` | `MilestoneNode[]` | Every milestone found across the backlog, sorted by name. |
| `tickets` | `TicketNode[]` | Every backlog item (active + paused + done), flat. |

## VisionDoc

| Field | Type | Notes |
|---|---|---|
| `id` | string | One of `specification`, `roadmap`, `gettingStarted`, `architectureDiagram`, `swarmFlowDiagram`. |
| `title` | string | Display title. |
| `kind` | `"markdown" \| "mermaid"` | How the PWA should render `content`. |
| `content` | string | Raw file content, unmodified. |

A document whose source file doesn't exist at the generated SHA is simply
absent from the array (not a null placeholder).

## MilestoneNode

| Field | Type | Notes |
|---|---|---|
| `milestone` | string | A milestone name, or `"unspecified"` for tickets with no `milestone:` field. |
| `tickets` | `MilestoneTicketSummary[]` | `{ id, title, status, priority? }` — status is folder-authoritative (`active`/`paused`/`done`), not the ticket's own possibly-stale YAML `status:` field. |

## TicketNode

| Field | Type | Notes |
|---|---|---|
| `id` | string | |
| `title` | string | |
| `status` | `"active" \| "paused" \| "done"` | Folder-authoritative. |
| `priority` | number (optional) | |
| `milestone` | string (optional) | Absent if the ticket has no milestone (it still appears under `milestones`' `"unspecified"` entry). |
| `description` | string (optional) | The ticket's prose `description:` field, trimmed. Absent if the ticket has none. |
| `scenarios` | `GherkinScenario[]` | Resolved acceptance scenarios, readable as scenario text. Empty array if the ticket has no acceptance field, or its `specs/features/*.feature` reference doesn't resolve. |

### GherkinScenario

| Field | Type | Notes |
|---|---|---|
| `name` | string | The text after `Scenario:` / `Scenario Outline:`. |
| `text` | string | The full scenario block (including its own `Scenario:` line and every step/Examples line), trimmed. |

## Acceptance resolution (both forms)

A ticket's `acceptance:` YAML field is one of two forms, both resolved into
the same `scenarios` shape:

- **File reference** (post-BL-111 convention): a single-line value ending in
  `.feature`, e.g. `specs/features/BL-100-thing.feature`. The generator
  reads that file (relative to the repo root at the generated SHA) and
  extracts its scenarios.
- **Inline Gherkin** (pre-BL-111 convention, still the majority of tickets
  as of this writing): a multi-line YAML block scalar (`acceptance: |`)
  containing Gherkin text directly. The generator extracts scenarios from
  that text directly, no file read involved.

Either form produces identical `GherkinScenario[]` shapes — the PWA never
needs to know which form a given ticket used.

## What is NOT in this file

- Live pipeline/agent state (that's `backlog.json`'s `board`/BridgeState,
  or the bridge's own live endpoints — this file is generated once per
  push, not live).
- Anything requiring write access or authentication — this is a pure,
  public, read-only projection (see `docs/backlog-dashboard-schema.md`'s
  own exposure note, which applies unchanged).
