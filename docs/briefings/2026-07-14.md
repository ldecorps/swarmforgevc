## Business features delivered

- **The phantom-revert mystery is solved (BL-373).** Every relaunch was silently `cp -R`-ing the launching checkout's `swarmforge/scripts/` and `swarmforge/profiles/` over every role worktree, clobbering any script fix merged onto a branch but not yet on `main`. Six roles independently re-diagnosed this as something else (hot-sync, the Operator, the tunnel) before the real cause was found — this closes that loop for good, though it will still re-fire on every relaunch until the underlying sync is itself rewritten, so it stays a "restore and log, don't re-investigate" item for now.
- **Three more control-plane failure modes found and fixed alongside it:** the tmux socket lived in `/tmp`, where anything could unlink it and permanently decapitate a live swarm (BL-367); `start-swarm.sh` left the tmux server in the launcher's own process group, so the whole swarm died with whoever started it (BL-372); and the generated systemd units couldn't actually start, silently defeating the crash-burst guard they claimed to provide (BL-366). Separately, **the swarm was never pushing `main` on its own** — hours of QA-approved, merged work could sit local-only, making a live swarm indistinguishable from a dead one from the outside (BL-356), and a zero-byte handoff file could be dispatched as a contentless task with the real commit silently lost (BL-365).
- **Front desk / Telegram reliability, round two.** Yesterday's own flagged defect — the starvation alarm arming itself on a failed send instead of a confirmed delivery — is fixed (BL-345). The topic-serialisation and lifecycle-reconciliation pair that was mid-fix and stalled in yesterday's briefing both landed (BL-329, BL-330). A reply could land in a different Telegram topic than the human asked in, making an answered question look like a dead swarm (BL-355); an answer typed in the wrong topic was silently lost and the human then nagged for something he'd already answered (BL-354); a blocked role holding no ticket was dropped instead of asked (BL-358); a reboot could silently kill the entire phone/Telegram surface with no boot unit to bring it back (BL-351); and BL-348 found that 26 of the swarm's Telegram topic records had no opening summary and 31 of 34 were never even committed — now complete and durable. BL-334, the restricted front-desk Operator previewed in yesterday's briefing as "in flight," formally closed today.

## Process hiccups

- **An engineering rule itself was wrong and it hard-aborted a production mutation run (BL-363).** The thin-wrapper rule's own prescribed fix — converting 32 CLI tests to `process.chdir()` for the in-process `main()` seam — was correct under a plain `vitest run` but fatal under Stryker's forced worker-thread pool, where `chdir()` throws and takes the whole mutation run down with it. The rule has been corrected (stub `process.cwd()`/`os.homedir()` instead), but it's a reminder that a rule proposal needs the same scrutiny as code — this one shipped, was followed literally, and broke a gate before anyone caught it.
- **The phantom-revert defect (BL-373) cost six separate role sessions before root cause was found**, each re-deriving the same wrong theories (hot-sync, the Operator, the tunnel) independently. The fix is landed but the underlying cause (`sync_worktree_scripts()`'s `cp -R`) still re-fires on every relaunch, so every role now has explicit instructions to restore-and-log rather than re-diagnose.
