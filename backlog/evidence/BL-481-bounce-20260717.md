# BL-481 QA bounce evidence — 2026-07-17

1. **Failing command** (exactly as run):
   ```
   bash swarmforge/scripts/test/test_operator_runtime_tick.sh
   ```
   (originally piped `2>&1 | tail -100`, which is what made the defect visible:
   the pipe never closed and the invocation hung well past a 120s timeout.)

2. **Commit hash tested**: `99debae2e7` (documenter's handoff to QA; the
   defect lives in `swarmforge/scripts/test/test_operator_runtime_tick.sh`,
   introduced at coder commit `7a21d75f5c` and still present after hardener
   commit `68d71b1829`).

3. **First error excerpt** — the script itself reports `ALL CHECKS PASSED`
   (exit 0), so the defect is a leaked process, not a failing assertion.
   Reproduced twice, deterministically, by running the file standalone:
   ```
   $ timeout 60 bash swarmforge/scripts/test/test_operator_runtime_tick.sh > out.log 2>&1
   $ echo exit:$?
   exit:0
   $ tail -3 out.log
   ok   - BL-481 live loop: pid-file is cleaned up on stop
   ok   - operator named 'Operator' (not a swarm agent)
   operator_runtime smoke: ALL CHECKS PASSED
   $ ps aux | grep 'sleep 300' | grep -v grep
   carillon  519686  0.0  0.0   3128  1028 ?        S    10:58   0:00 sleep 300
   ```
   A `sleep 300` process from the test's own fixture is still alive several
   seconds after the script exited with all checks green. (First reproduction,
   under the original piped invocation, actually blocked a `tail -100`
   consumer for the full remainder of the 300s sleep before it was manually
   killed — confirmed via `pstree`/`ps --ppid` showing the piping shell still
   waiting on EOF from the pipe with no live producer besides the orphaned
   `sleep`.)

4. **Failure class**: `behavior` — the test's actual runtime behavior
   (leaving a spawned background process alive after the script exits) does
   not match the engineering-article contract every other section of this
   same file already follows (spawn under `FD_PIDS`, then `cleanup_fd_pids`
   before moving on / disarming the trap). This is not a compile or
   assertion failure; the checks all pass.

5. **Expected vs observed**: Expected — every `sleep 300 &` (or other
   backgrounded pid) the test spawns to simulate "an Operator/loop is
   running" is killed before the script exits, exactly as the pre-existing
   BL-334 section does (`cleanup_fd_pids; trap - EXIT` at
   `test_operator_runtime_tick.sh:936-937`). Observed — the BL-481 section
   added by the coder at `test_operator_runtime_tick.sh:981`
   (`sleep 300 & op_pid=$!; FD_PIDS+=("$op_pid"); ...` in the "guard: an
   already-running full Operator blocks the fast-path launch too" check)
   appends to the SAME `FD_PIDS` array, but by that point in the file the
   `trap cleanup_fd_pids EXIT` armed at line 823 has already been disarmed
   (`trap - EXIT` at line 937, from the earlier BL-334 section) and never
   re-armed — so `op_pid` is neither killed by a trap nor by any explicit
   `cleanup_fd_pids` call added after it, and survives the whole script.
   (The hardener's own later addition, `loop_pid` at line ~1014, does NOT
   leak — it self-terminates via the live loop's own stop-file test — so
   this is scoped to the one `op_pid` at line 981.)

## Root cause
`test_operator_runtime_tick.sh`'s existing convention is: arm
`trap cleanup_fd_pids EXIT` once (line 823), let several sections push onto
`FD_PIDS`, then explicitly call `cleanup_fd_pids` followed by `trap - EXIT`
once that block of sections is done (lines 936-937) — because the file reuses
this pattern more than once and does not want a stale trap firing on
processes a LATER, unrelated section did not spawn. The BL-481 coder
commit added a NEW `FD_PIDS`-consuming section immediately after that
disarm point, reusing the array/function name without reinstating the
cleanup — an easy mistake given the convention is "call cleanup_fd_pids at
the end of your block," which the new block simply never does.

## Suggested fix
Add `cleanup_fd_pids` (and, if intended to stay reusable for any later
section, re-arm `trap cleanup_fd_pids EXIT` first) after the BL-481
`op_pid` guard test at line ~984 (before `rm -rf "$F"`), and again after the
live-loop section once it's done (harmless belt-and-suspenders there, since
that pid already self-terminates) — matching the exact convention the
BL-334 section already established two blocks above.
