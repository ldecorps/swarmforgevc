# BL-100 scope gap — cost-05/cost-06 depend on infrastructure that isn't
# built yet, and BL-100's own depends_on doesn't list it

## What BL-100 assumes exists

BL-100's acceptance (`specs/features/BL-100-token-cost-resource-telemetry.feature`)
has 7 scenarios. Two of them (`cost-05`, `cost-06`) assume two other tickets'
deliverables are already in place to extend:

- `cost-05` ("briefing carries the cost & health paragraph"): assumes the
  BL-099 daily briefing already has a working content contract to add a
  paragraph to.
- `cost-06` ("phone sees daily figures via the committed briefing"): assumes
  a `docs/briefings/<date>.json` sidecar convention and a GitHub Action that
  parses it into `backlog.json` (BL-097's scope) already exist.

BL-100's own `depends_on` field lists only `BL-096 (trend framework), BL-098
(telemetry log family + holding periods)` — it does not list BL-097 or
BL-099, even though cost-05/06 read as hard prerequisites on both.

## What actually exists (checked before writing this)

- **BL-099** (`backlog/done/M6-remote-chat/BL-099-coordinator-daily-briefing.yaml`,
  status done): its own content contract (BUSINESS FEATURES DELIVERED /
  trends / agent business / optimizer recs / process hiccups) was never
  copied into `swarmforge/roles/coordinator.prompt` - the prompt only
  mentions "the daily briefing" in passing (2 hits, both inside the "Swarm
  Optimizer" section), with no structured briefing-writing instructions. The
  briefing is composed by the coordinator LLM from a tmux nudge
  (`extension/src/extension.ts`'s `startOrRestartDailyBriefing`), not by a
  deterministic script - there is no code path to extend with a new
  paragraph. `docs/briefings/` has no committed file in this repo today; the
  one historical attempt (`docs/briefings/2026-07-07.md`, commit `b0a0065`)
  was reverted same-day (`aeb7dd1`) and didn't follow BL-099's own content
  contract anyway (a generic per-agent status stub, not a business-first
  coordinator briefing). The `docs/briefings/<date>.json` sidecar BL-100
  wants to add a metrics block to does not exist anywhere - no schema, no
  example, no code.
- **BL-097** (`backlog/paused/BL-097-backlog-dashboard-pwa-pages.yaml`,
  status todo): this is the ticket that's supposed to build the Action +
  `backlog.json` generator BL-100's cost-06 wants to add fields to. It's
  unimplemented - no workflow, no generator script, no `backlog.json`
  example, no `schema_version` field anywhere in the repo. The repo's one
  GitHub Action (`.github/workflows/swarm-intake.yml`) does something
  unrelated (issue-to-backlog-item intake) and never touches `backlog.json`.
  The PWA phone consumer BL-100 mentions doesn't exist either.

## Failure class
`scope`

## Recommendation

Same pattern as BL-096's own slicing (which explicitly unblocked BL-100's
trend-framework dependency without waiting on BL-094's larger UI build):
slice BL-100 into a computation/telemetry layer (cost-01, 02, 03, 04, 07 -
transcript parser, cost table, resource sampler, all genuinely buildable
now against BL-096/BL-098) that lands first, and defer cost-05/cost-06 to a
follow-up ticket whose `depends_on` correctly includes BL-097 and BL-099's
*actual* implementation (not just the done ticket ID), so that follow-up
isn't picked up before its real prerequisites exist.

Proceeding now with the computation/telemetry layer (cost-01/02/03/04/07)
while this note is in flight, since that portion has no such gap and is
independently valuable per the ticket's own "operator wants this generously,
prune later" framing. Not implementing cost-05/cost-06 without specifier
direction on how to handle the missing BL-099/BL-097 foundations - inventing
the sidecar schema and retrofitting coordinator.prompt's briefing content
contract unilaterally would be deciding scope that belongs to
specifier/architect, not coder.
