BUG (raw intake, from coordinator observation 2026-07-06 ~19:05 local):
BL-137's wake-up escalation path was silently removed by a later commit,
reopening BL-137's original symptom for the unrecoverable case.

What was observed live:
handoffd.log shows the exact BL-137 failure signature recurring tonight,
after BL-137 was already closed done:

  18:44:35Z notify-delivery-failed swarmforge-QA "pane already held
    undelivered input and it still would not submit"
  18:47:52Z notify-delivery-failed swarmforge-specifier  (same)
  18:47:53Z notify-delivery-failed swarmforge-coder      (same)
  18:47:56Z notify-delivery-failed swarmforge-documenter (same)

In every case the handoff file was still marked "delivered" and filed into
the role's inbox/new/ even though the wake keystroke never actually
submitted. All four roles then sat genuinely idle (confirmed via ~0% CPU
on their `claude` processes) with real work waiting and no one told —
i.e. exactly BL-137's "idle agents stall until a manual kick" symptom.
Root-caused live by the coordinator: sending Ctrl-C, Ctrl-U, then the
wake text + Enter manually into each pane immediately unstuck all four.

Root cause (why BL-137's fix regressed):
BL-137's accepted design had two layers: (1) a harder clear-input +
retry-and-confirm recovery, and (2) an escalation of last resort —
"if a pane still will not submit, treat the agent as wedged and respawn
its pane" (BL-137 acceptance scenario escalate-02).

A later commit, 5ef8dd9 "Disable automatic chaser pane respawn", changed
ChaserMonitor's `triggerRespawn` callback (extension/src/extension.ts) from
calling `respawnAgent(targetPath, role)` to only calling
`setStuckEscalation(role, true)`. That change was made for a good reason
(BL-137-follow-up: ChaserMonitor.triggerRespawn had typed
`bash .../coordinator.sh` into a LIVE, busy pane mid-turn — a real
incident) — but it removed the only automatic recovery for a pane that
handoffd's own clear+retry logic could not unstick, and replaced it with
a flag (`setStuckEscalation`) that nothing currently reads or acts on.
So today: transient stuck-input cases still recover fine, but the
genuinely wedged case (handoffd logs notify-delivery-failed) now has NO
automatic recovery at all — same as pre-BL-137.

What's needed (for the specifier to turn into a proper spec):
- Escalation must be re-added, but must not repeat the mid-turn-injection
  incident that motivated disabling it. The fix should distinguish
  "genuinely wedged" (idle, ~0% CPU, notify-delivery-failed after
  handoffd's own retries) from "busy/mid-turn" (active CPU, no
  notify-delivery-failed) BEFORE ever considering a respawn — i.e. the
  precheck that was added for the coordinator (esc-to-interrupt /
  liveness check before respawn injection, per
  architect_bl137_respawn_guard_reviewed) needs to also gate this
  escalation path, not just the earlier misfiring one.
- Something needs to actually consume `escalatedStuckRoles` /
  `setStuckEscalation` state (surface it, or act on it) — right now it is
  written but nothing reads it, which is how this went unnoticed.
